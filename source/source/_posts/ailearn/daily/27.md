今天我们来学习 **AI Agent 的设计模式 —— Orchestrator-Workers（编排器-工作者）**

> 一句话核心：**Orchestrator-Workers** 就像一个项目经理带团队——Orchestrator（编排器）负责整体规划和任务分配，多个 Workers（工作者）并行处理各自的部分，最后 Orchestrator 整合所有结果。

说实话，这个模式太像我们程序员的工作日常了。你有没有经历过这样的场景：

项目来了，技术负责人先分析需求，然后把任务拆成前端、后端、数据库几个模块，分给不同的开发同学。大家各自写代码，开发过程中遇到问题再同步给负责人。等所有模块都完成后，负责人把代码合并、跑通联调，最后交付。

这就是 Orchestrator-Workers 的核心思想：**一个"大脑"负责统筹协调，多个"执行者"并行工作**。

它和前面讲的 Pure Parallelization（纯并行）有什么区别？纯并行是拆了就跑，各自独立。而 Orchestrator-Workers 强调**中心化的协调和动态调度**——Orchestrator 不仅分配任务，还监控进度、处理依赖、整合结果，甚至根据中间结果动态调整后续任务。

它的核心价值在于**处理复杂依赖关系、支持动态任务调度**——适合那些任务之间有千丝万缕联系，需要灵活变通的场景。

# 是什么

老规矩，先看图解：

![Orchestrator-Workers 工作原理](/assets/img/ailearn/daily/34/1.png)

从图里能看到，Orchestrator-Workers 有三个核心角色：

- **Orchestrator（编排器）**：团队的"项目经理"，负责分析任务、拆解子任务、分配给 Workers、监控执行、处理异常、最后整合结果
- **Workers（工作者）**：团队的"开发成员"，各自处理被分配的子任务，可以并行工作
- **Communication Channel（通信通道）**：Orchestrator 和 Workers 之间的信息通路，Worker 执行过程中可以随时反馈，Orchestrator 可以根据反馈调整策略

整个流程是这样的：

1. **接收任务和分析**：Orchestrator 收到一个复杂任务，比如"帮我做一个电商系统"
2. **拆解和分配**：Orchestrator 将任务拆分为：
   - 用户模块（Worker A 负责）
   - 商品模块（Worker B 负责）
   - 订单模块（Worker C 负责）
3. **并行执行+动态协调**：Workers 开始各自开发，过程中：
   - Worker A 发现用户表设计需要订单表 ID，通知 Orchestrator
   - Orchestrator 协调 Worker C 优先完成订单表接口
   - Worker A 拿到接口后继续开发
4. **结果整合**：所有模块完成后，Orchestrator 负责代码合并、接口联调、测试验证

举个我折腾的真实例子：

有次我让 AI 完成一个复杂的数据分析任务，涉及销售数据、用户行为、竞品对比三个维度。如果用纯并行，三个 Worker 各自为战，可能会得出相互矛盾的结论。

用 Orchestrator-Workers 后：

- Orchestrator 先定分析框架和指标
- 三个 Worker 同时跑各自的数据
- Worker B 发现用户留存率和 Worker A 的销售数据有冲突，立即反馈
- Orchestrator 协调：让 Worker A 验证数据准确性
- 最终 Orchestrator 整合三份报告，确保数据口径一致

这就是 Orchestrator-Workers 的强大之处：**不是简单的一锤子买卖，而是持续的协调和优化**。

## 核心架构

Orchestrator-Workers 的实现思路：

```python
class OrchestratorWorkersAgent:
    def __init__(self):
        self.orchestrator = Orchestrator()
        self.workers = {name: Worker(name) for name in worker_names}

    def run(self, task):
        # Step 1: Orchestrator 分析任务并制定执行计划
        execution_plan = self.orchestrator.plan(task)

        # Step 2: 按照计划启动 Workers 并行执行
        results = {}
        for subtask in execution_plan.subtasks:
            worker = self.workers[subtask.assigned_worker]
            result = worker.execute(subtask)
            results[subtask.id] = result

            # Step 2.5: 动态检查是否需要调整计划
            if self.orchestrator.need_adjustment(result):
                execution_plan = self.orchestrator.adjust_plan(
                    execution_plan, result
                )

        # Step 3: Orchestrator 整合所有结果
        final_result = self.orchestrator.aggregate(execution_plan, results)

        return final_result
```

关键点在于：

- **Orchestrator 要有全局视野**：不仅分配任务，还要理解任务间的依赖关系
- **支持动态调整**：执行过程中发现问题，要能重新规划
- **Workers 可以反馈信息**：不只是单向的"分配给 Workers"，Workers 也能告诉 Orchestrator "这个不行，得改"
- **结果要统一整合**：不是简单拼接，而是 Orchestrator 基于全局理解做合并

## 实战代码

来看一个完整的实现，做一个多维度数据分析助手：

```python
import json
import asyncio
from typing import List, Dict, Any, Optional, Callable
from dataclasses import dataclass, field
from concurrent.futures import ThreadPoolExecutor, as_completed
from enum import Enum
import time

# 模拟 LLM 客户端
class MockLLM:
    def generate(self, prompt: str) -> str:
        """模拟 LLM 生成响应"""
        return f"[LLM Response] 基于以下输入分析：{prompt[:100]}..."

llm = MockLLM()

@dataclass
class SubTask:
    """子任务定义"""
    task_id: str
    name: str
    description: str
    worker_type: str  # 指定哪种类型的 Worker 处理
    dependencies: List[str] = field(default_factory=list)  # 依赖的其他子任务ID
    input_data: Any = None
    status: str = "pending"  # pending, running, completed, failed
    result: Any = None

@dataclass
class ExecutionPlan:
    """执行计划"""
    goal: str
    subtasks: List[SubTask]
    overall_strategy: str

class Worker:
    """工作者：执行具体子任务"""

    def __init__(self, worker_id: str, worker_type: str):
        self.worker_id = worker_id
        self.worker_type = worker_type
        self.capabilities = self._get_capabilities(worker_type)

    def _get_capabilities(self, worker_type: str) -> List[str]:
        """定义不同类型 Worker 的能力"""
        capabilities_map = {
            "data_analyst": ["数据分析", "统计分析", "趋势识别"],
            "user_researcher": ["用户调研", "行为分析", "满意度分析"],
            "competitor_analyst": ["竞品分析", "市场对比", "SWOT分析"],
            "integrator": ["报告整合", "一致性检查", "冲突解决"]
        }
        return capabilities_map.get(worker_type, [])

    def execute(self, subtask: SubTask) -> Dict[str, Any]:
        """执行子任务"""
        print(f"  🔄 Worker-{self.worker_id}({self.worker_type}) 执行任务: {subtask.name}")
        start_time = time.time()

        try:
            # 根据 Worker 类型和任务内容生成模拟结果
            result = self._process_task(subtask)
            execution_time = time.time() - start_time

            print(f"  ✅ Worker-{self.worker_id} 完成 (耗时 {execution_time:.2f}s)")

            return {
                "task_id": subtask.task_id,
                "worker_id": self.worker_id,
                "success": True,
                "result": result,
                "execution_time": execution_time,
                "feedback": None
            }
        except Exception as e:
            execution_time = time.time() - start_time
            print(f"  ❌ Worker-{self.worker_id} 失败: {str(e)}")
            return {
                "task_id": subtask.task_id,
                "worker_id": self.worker_id,
                "success": False,
                "error": str(e),
                "execution_time": execution_time
            }

    def _process_task(self, subtask: SubTask) -> str:
        """处理具体任务（模拟）"""
        # 实际项目中这里会调用 LLM 或执行具体逻辑
        prompt = f"""
        你是一名专业的{self.worker_type}。
        任务：{subtask.name}
        描述：{subtask.description}
        输入数据：{subtask.input_data}

        请给出专业的分析结果。
        """

        # 模拟不同 Worker 的返回
        if self.worker_type == "data_analyst":
            return f"销售数据分析结果：Q1-Q4 季度销售额同比增长 15%，主要增长点来自新品类。{llm.generate(prompt)}"
        elif self.worker_type == "user_researcher":
            return f"用户行为分析：活跃用户月均增长 8%，用户留存率稳定在 65%。{llm.generate(prompt)}"
        elif self.worker_type == "competitor_analyst":
            return f"竞品对比分析：在价格方面我们有 10% 优势，但功能丰富度落后竞品。{llm.generate(prompt)}"
        elif self.worker_type == "integrator":
            return f"整合报告：综合分析显示，销售增长主要得益于用户留存提升和新用户获取的平衡。{llm.generate(prompt)}"
        else:
            return llm.generate(prompt)

class Orchestrator:
    """编排器：负责整体协调"""

    def __init__(self):
        self.workers: Dict[str, Worker] = {}
        self.execution_history: List[Dict] = []

    def register_worker(self, worker_id: str, worker: Worker):
        """注册 Worker"""
        self.workers[worker_id] = worker
        print(f"  📝 注册 Worker: {worker_id} ({worker.worker_type})")

    def plan(self, goal: str, input_data: Any) -> ExecutionPlan:
        """Step 1: 制定执行计划"""
        print(f"\n📋 Step 1: Orchestrator 制定执行计划")
        print(f"  目标：{goal}")

        # 分析任务并拆解
        # 实际项目中这里会调用 LLM 进行分析
        subtasks = [
            SubTask(
                task_id="task_1",
                name="销售数据分析",
                description="分析过去一年的销售数据，识别增长趋势",
                worker_type="data_analyst",
                input_data=input_data.get("sales_data")
            ),
            SubTask(
                task_id="task_2",
                name="用户行为分析",
                description="分析用户活跃度和留存率",
                worker_type="user_researcher",
                input_data=input_data.get("user_data")
            ),
            SubTask(
                task_id="task_3",
                name="竞品对比分析",
                description="对比主要竞品的优劣势",
                worker_type="competitor_analyst",
                input_data=input_data.get("competitor_data")
            ),
            SubTask(
                task_id="task_4",
                name="数据一致性检查",
                description="检查各分析结果的一致性",
                worker_type="integrator",
                dependencies=["task_1", "task_2", "task_3"]
            ),
            SubTask(
                task_id="task_5",
                name="生成最终报告",
                description="整合所有分析生成综合报告",
                worker_type="integrator",
                dependencies=["task_4"]
            )
        ]

        plan = ExecutionPlan(
            goal=goal,
            subtasks=subtasks,
            overall_strategy="并行执行独立分析任务，然后统一检查一致性，最后整合报告"
        )

        print(f"  ✓ 生成 {len(subtasks)} 个子任务")
        for st in subtasks:
            deps = f" [依赖: {st.dependencies}]" if st.dependencies else ""
            print(f"    - {st.name} (分配给: {st.worker_type}){deps}")

        return plan

    def execute_plan(self, plan: ExecutionPlan) -> Dict[str, Any]:
        """Step 2: 执行计划"""
        print(f"\n⚡ Step 2: 执行任务计划")

        results = {}
        completed_tasks = set()

        # 分组执行：先执行独立任务，再执行依赖任务
        independent_tasks = [t for t in plan.subtasks if not t.dependencies]
        dependent_tasks = [t for t in plan.subtasks if t.dependencies]

        # Phase 1: 并行执行独立任务
        print(f"\n  Phase 1: 并行执行 {len(independent_tasks)} 个独立任务")
        with ThreadPoolExecutor(max_workers=3) as executor:
            futures = {}
            for subtask in independent_tasks:
                # 分配合适的 Worker
                worker = self._get_available_worker(subtask.worker_type)
                future = executor.submit(worker.execute, subtask)
                futures[future] = subtask

            for future in as_completed(futures):
                subtask = futures[future]
                result = future.result()
                results[subtask.task_id] = result
                completed_tasks.add(subtask.task_id)

        # Phase 2: 顺序执行依赖任务
        print(f"\n  Phase 2: 顺序执行 {len(dependent_tasks)} 个依赖任务")
        for subtask in dependent_tasks:
            # 检查依赖是否完成
            pending_deps = [d for d in subtask.dependencies if d not in completed_tasks]
            if pending_deps:
                print(f"    ⏳ 等待依赖: {pending_deps}")
                continue

            # 动态检查是否需要调整
            if self.need_adjustment(results, subtask):
                print(f"    🔄 基于中间结果调整任务: {subtask.name}")
                subtask = self.adjust_task(subtask, results)

            worker = self._get_available_worker(subtask.worker_type)
            result = worker.execute(subtask)
            results[subtask.task_id] = result
            completed_tasks.add(subtask.task_id)

        return results

    def _get_available_worker(self, worker_type: str) -> Worker:
        """获取指定类型的可用 Worker"""
        for worker_id, worker in self.workers.items():
            if worker.worker_type == worker_type:
                return worker
        raise ValueError(f"没有可用的 {worker_type} 类型 Worker")

    def need_adjustment(self, current_results: Dict, next_task: SubTask) -> bool:
        """动态判断是否需要调整计划"""
        # 检查是否有异常结果需要处理
        for task_id, result in current_results.items():
            if not result.get("success", True):
                return True

            # 检查数据一致性
            result_text = str(result.get("result", ""))
            if "冲突" in result_text or "矛盾" in result_text:
                return True

        return False

    def adjust_task(self, task: SubTask, current_results: Dict) -> SubTask:
        """动态调整任务"""
        # 增加更详细的检查说明
        task.description += " [调整：增加数据验证和冲突检查]"
        print(f"    ✓ 任务已调整: {task.name}")
        return task

    def aggregate(self, plan: ExecutionPlan, results: Dict[str, Any]) -> str:
        """Step 3: 整合结果"""
        print(f"\n📊 Step 3: 整合最终结果")

        # 统计信息
        success_count = sum(1 for r in results.values() if r.get("success"))
        total_count = len(results)

        print(f"  任务完成度: {success_count}/{total_count}")

        # 收集所有结果
        all_results = []
        for task_id, result in results.items():
            if result.get("success"):
                all_results.append({
                    "task_id": task_id,
                    "result": result.get("result"),
                    "worker_id": result.get("worker_id")
                })

        # 实际项目中这里会调用 LLM 做智能整合
        final_report = f"""
# 综合分析报告

## 执行概况
- 目标：{plan.goal}
- 策略：{plan.overall_strategy}
- 任务完成：{success_count}/{total_count}

## 各维度分析结果

"""
        for item in all_results:
            final_report += f"### {item['task_id']}\n{item['result']}\n\n"

        final_report += """
## 综合结论
基于多维度分析，建议：
1. 继续保持销售增长势头
2. 关注用户留存率优化
3. 加强产品功能投入以应对竞品竞争
"""

        print(f"  ✓ 报告生成完成")
        return final_report

    def run(self, goal: str, input_data: Any) -> str:
        """完整流程入口"""
        print(f"\n🎯 Orchestrator 启动")
        print("=" * 60)

        start_time = time.time()

        # Step 1: 制定计划
        plan = self.plan(goal, input_data)

        # Step 2: 执行计划
        results = self.execute_plan(plan)

        # Step 3: 整合结果
        final_result = self.aggregate(plan, results)

        total_time = time.time() - start_time
        print(f"\n⏱️ 总耗时: {total_time:.2f} 秒")

        return final_result
```

运行这段代码，你会看到：

1. **Orchestrator 制定计划**：分析任务、拆解子任务、识别依赖关系
2. **Phase 1 并行执行**：独立任务同时分配给多个 Workers
3. **Phase 2 顺序执行**：依赖任务按顺序等待前置任务完成
4. **动态调整**：当 Worker 反馈问题，Orchestrator 自动调整后续任务
5. **结果整合**：统一汇总所有 Worker 的输出

## 几个实用技巧

- **Worker 类型设计**：根据业务场景定义不同类型的 Worker（分析师、研究员、集成器等）
- **依赖关系建模**：用 DAG（有向无环图）来建模任务间的依赖关系
- **反馈机制**：Worker 不仅要返回结果，还要能返回状态、异常、建议
- **超时和重试**：对每个 Worker 执行设置超时和失败重试机制
- **资源池管理**：如果 Workers 是昂贵的资源（如 GPU），要考虑复用和回收
- **中间状态持久化**：复杂任务要保存中间结果，支持断点续传

# ❄️ 冷知识

1. **Orchestrator-Workers 的工程化对应**：这种模式在软件工程中有个专业名词叫"主从架构"（Master-Slave Architecture）。Redis 的主从复制、Kubernetes 的 Master-Node 架构都是这个思想的工程落地。

2. **和 MapReduce 的结合**：你可以把 Orchestrator-Workers 看作是 MapReduce 的升级版。MapReduce 的 Mapper 就像 Workers，Reducer 就像 Orchestrator 的结果整合，但 Orchestrator-Workers 增加了**动态调度**和**双向通信**的能力。

3. **LangChain 的实现**：在 LangChain 中，有一个类似的实现叫 `Multi-Agent`，其中的 `Supervisor` 就是 Orchestrator 的角色，而 `Worker` Agents 就是实际执行者。这种模式也被称为"监督者模式"。

4. **成本控制的挑战**：Orchestrator-Workers 的模式虽然强大，但要注意成本。因为 Orchestrator 要反复调用 LLM 来做计划、做调整、做整合，如果任务复杂，Orchestrator 本身可能成为瓶颈和成本大户。

5. **AutoGPT 的启发**：早期爆火的 AutoGPT 就是基于类似的架构——一个主循环（Orchestrator）不断拆解任务、分配给各种工具（Workers），然后根据结果调整下一步行动。虽然后来被证明实用性有限，但这种架构思想影响深远。
